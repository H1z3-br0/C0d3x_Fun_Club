# ТЗ: CTF Multi-Agent Swarm System

> **Для Claude Code:** Перед тем как начать реализацию, внимательно прочитай это ТЗ целиком. Если что-то непонятно, неоднозначно или требует уточнения — задай все вопросы сразу, одним списком. Не начинай писать код до получения ответов на вопросы.

---

## Что это

Система автоматически решает задачи CTF соревнований (jeopardy формат, до 12 часов непрерывной работы). Все агенты — языковые модели, вызываемые через HTTP API.

---

## Инфраструктура

Используется **CLIProxyAPI** (https://github.com/router-for-me/CLIProxyAPI) — Go-сервер, который оборачивает CLI-инструменты (Claude Code, OpenAI Codex) в OpenAI-совместимый API. Сервер запускается отдельно до старта системы.

Все вызовы агентов идут через стандартный OpenAI Python SDK на `http://localhost:8080/v1`.

---

## Агенты и их роли

### CC1 — Master (Claude Code Pro, 1 аккаунт)
Единственный агент, который **думает**. Не запускает инструменты напрямую.
- Анализирует CTF таск, генерирует список конкретных гипотез для проверки
- Раздаёт гипотезы воркерам
- Получает результаты, оценивает прогресс
- Решает убить ли воркера, сделать ли pivot, углубить ли перспективное направление
- При исчерпании своего лимита — пишет compressed summary и передаёт роль

### CC2 — Support (Claude Code Pro, 1 аккаунт)
Работает параллельно с CC1, не мешает его контексту.
- При новом таске — ищет похожие решения в памяти, передаёт контекст CC1
- Ищет релевантные CVE и техники
- После нахождения флага — верифицирует и генерирует writeup
- Сохраняет решение в долгосрочную память
- В MULTI режиме — мониторит CTFd scoreboard и обновляет приоритеты тасков

### Smart Reserve — S1, S2, S3, S4 (Codex, 4 аккаунта)
Резервные агенты для замены CC1/CC2 когда те исчерпают лимиты.
- S1 → резерв CC1, S2 → резерв CC2
- S3 → резерв S1, S4 → резерв S2
- Работают по жёстким промптам (без творчества), только по готовым гипотезам

### Executor Pool (Codex, 20 аккаунтов)
Рабочие лошадки. Каждый получает одну конкретную гипотезу + набор инструментов.
- Выполняют инструменты, пишут скрипты, запускают команды
- Не принимают стратегических решений
- Отчитываются структурированным JSON
- Не знают о существовании других воркеров

## Ключевые принципы

**Гипотезы, не категории.** CC1 не классифицирует таск как "web" или "pwn". Он генерирует конкретные проверяемые гипотезы: "попробовать heap UAF через tcache poisoning", "проверить JWT alg=none", "запустить angr на функцию check_password". Категория используется только как подсказка для генерации гипотез, не для роутинга.

**Параллельность.** 10–15 воркеров одновременно проверяют разные гипотезы. Остальные 5–10 — в резерве для немедленного pivot.

**Pivot без задержки.** Воркер застрял → убить и заменить немедленно. Все зашли в тупик → CC1 генерирует новые гипотезы. Один нашёл что-то важное → убить остальных, сфокусировать всё на этом.

**Memory как позвоночник.** После каждого решённого таска — сохранить технику, инсайт, инструменты. При следующем таске — искать похожее. Система умнеет с каждым соревнованием.

---

## Ротация при исчерпании лимитов

При ~80% лимита CC1:
1. CC1 пишет compressed summary текущего состояния (активные гипотезы, находки, что не сработало)
2. Summary сохраняется в файл
3. S1 загружает summary и продолжает как новый Master

Фазы за 12-часовой CTF:
- Фаза 1 (0–2ч): CC1 + CC2 живые
- Фаза 2 (2–6ч): S1 → Master, S2 → Support
- Фаза 3 (6–12ч): S3 → Master, S4 → Support

Исполнители (executor pool) при ротации **не прерываются**.

---

## Memory

Долгосрочное хранилище решений. После каждого таска сохраняется:
- название таска, CTF, категория
- техники и инструменты которые сработали
- техники которые НЕ сработали
- ключевой инсайт (одно предложение)
- флаг

При новом таске — полнотекстовый поиск по похожим, передать контекст CC1.

Реализация на усмотрение (SQLite + FTS5 — хороший вариант).

## Изолированное выполнение

Все команды воркеров выполняются в Docker контейнере с CTF инструментами:
pwntools, gdb, binwalk, z3, angr, hashcat, radare2, volatility, scapy, pycryptodome.

Сеть в контейнере отключена по умолчанию (можно включить для web тасков).

## Входные данные таска

Система читает из `--task-dir`:
- Все файлы рекурсивно (бинари, исходники, изображения, архивы)
- `task.txt` или `README.*` если есть — как описание и контекст
- `task.yaml` если есть — структурированные метаданные

Формат `task.yaml` (всё опционально):
```yaml
name: "heap_challenge"
points: 500
description: "классический heap exploit"
url: "https://ctf.example.com/challenge"
host: "ctf.example.com"
port: 1337
```

`challenge_id` указывать не нужно — система находит таск на CTFd автоматически по названию.

Если `task.yaml` / `task.txt` отсутствуют — CC2 анализирует содержимое директории и составляет описание автоматически.

**Пользователь не обязан стандартизировать входные данные. Минимум — просто папка с файлами.**

## Режимы работы

### SINGLE (по умолчанию)
```bash
python main.py --task-dir ./one_task/ --flag-format "CTF{...}"
```
Все ресурсы направлены на один таск.

### MULTI
```bash
python main.py \
  --mode multi \
  --ctfd-url https://ctf.example.com \
  --ctfd-token ctfd_abc123... \
  --flag-format "CTF{...}"
```

В MULTI режиме система:
1. Забирает список всех нерешённых тасков с CTFd через API
2. Скачивает файлы каждого таска
3. Приоритизирует таски — формула на усмотрение реализации, учитывает очки и число команд решивших таск
4. Распределяет воркеров между тасками динамически (приоритетный таск получает больше воркеров)
5. При решении таска — его воркеры немедленно перераспределяются на остальные
6. CC2 периодически обновляет приоритеты по scoreboard

CTFd API используется **только для чтения**:
```
GET /api/v1/challenges             — список тасков + поиск по названию
GET /api/v1/challenges/{id}        — описание таска
GET /api/v1/challenges/{id}/files  — файлы таска
GET /api/v1/scoreboard             — для приоритизации
```

Аутентификация на CTFd — два варианта (указать один):
- `--ctfd-token` — API токен из Settings → Account → API Key (предпочтительно)
- `--ctfd-session` — session cookie из браузера (fallback для нестандартных CTFd форков)

## Флаги — только вывод, без автосабмита

Система **никогда не сабмитит флаги автоматически**. При нахождении флага:

1. Вывести флаг в stdout максимально заметно
2. Указать: название таска, уверенность, путь к writeup
3. Продолжить работу над остальными тасками (в MULTI режиме)

Пример вывода:
```
════════════════════════════════════════
  ФЛАГ НАЙДЕН: heap_challenge
  CTF{h34p_m4st3r_0f_tcach3_1337}
  Уверенность: 100% (верифицирован)
  Writeup: workspace/artifacts/heap_challenge/writeup.md
════════════════════════════════════════
```

Пользователь сабмитит флаг вручную на платформе.

---

## Формат флага

Задаётся обязательно перед запуском — через аргумент `--flag-format` или в `config.yaml`.

Пользователь указывает человекочитаемый пример формата, система сама строит из него regex:

```bash
--flag-format "CTF{...}"      # → regex: CTF\{[^}]+\}
--flag-format "flag{...}"     # → regex: flag\{[^}]+\}
--flag-format "Hero{...}"     # → regex: Hero\{[^}]+\}
```

В `config.yaml`:
```yaml
ctf:
  flag_format: "CTF{...}"
```

Аргумент командной строки перекрывает значение из конфига.

## Human-in-the-loop

Система поддерживает прерывание и корректировку в реальном времени через stdin пока запущена.

**Ctrl+C** — останавливает текущий шаг чисто (без потери состояния), предлагает меню:
```
[1] продолжить
[2] дать указание CC1
[3] сменить активный таск
[4] выход с сохранением состояния
```

**Команды во время работы:**

| Команда | Действие |
|---|---|
| `/hint "текст"` | Добавить подсказку в контекст CC1 без прерывания воркеров |
| `/interrupt "текст"` | Остановить всё, передать сообщение CC1, ждать нового плана |
| `/kill cx-04` | Убить конкретного воркера |
| `/status` | Показать состояние всех агентов и воркеров |
| `/skip` | Отказаться от текущего таска, перейти к следующему (MULTI) |

`/hint` не прерывает работу — CC1 учтёт подсказку на следующем цикле оценки.
`/interrupt` останавливает воркеров и ждёт пока CC1 пересмотрит план.

## Конфигурация

Один файл `config.yaml`:
```yaml
cliproxyapi:
  base_url: "http://localhost:8080/v1"

accounts:
  cc_master:
    type: claude-code
    emails: ["cc1@gmail.com"]
  cc_support:
    type: claude-code
    emails: ["cc2@gmail.com"]
  smart_reserve:
    type: codex
    emails:
      - "reserve1@gmail.com"
      - "reserve2@gmail.com"
      - "reserve3@gmail.com"
      - "reserve4@gmail.com"
  executors:
    type: codex
    emails:
      - "worker01@gmail.com"
      # ... до worker20

limits:
  worker_max_steps: 15
  worker_timeout_seconds: 300
  max_parallel_workers: 15
  cc_usage_warning_pct: 0.80
  rate_limit_backoff_seconds: 65

sandbox:
  enabled: true
  network_disabled: true

ctf:
  flag_format: "CTF{...}"     # перекрывается аргументом --flag-format
```

## Запуск

```bash
# Одна задача
python main.py --task-dir ./pwn_challenge/ --flag-format "CTF{...}"

# Одна задача с продолжением после сбоя
python main.py --task-dir ./pwn_challenge/ --flag-format "CTF{...}" --resume

# Всё CTF соревнование
python main.py \
  --mode multi \
  --ctfd-url https://ctf.example.com \
  --ctfd-token ctfd_abc123... \
  --flag-format "CTF{...}"
```
